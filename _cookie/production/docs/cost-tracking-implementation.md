# Cost Tracking Implementation

> Three-stage cost system: Estimation (before API calls) → Real-Time Tracking (during generation) → Final Report (after delivery).

## Overview

Every API call in COOKIE has a cost. The cost system ensures users always know what they're spending, can approve budgets before generation, and get detailed reports after delivery.

## Stage 1: Cost Estimation

Executed by **Task T2** (`cost-estimate.xml`) at the end of pre-production, before any API calls.

### Process

1. Parse the SSD (scene-spec.yaml)
2. Count all API operations per scene:
   - VEO calls (text-to-video, image-to-video, interpolation)
   - Nano Banana calls (character refs, first/last frames, style images)
   - ElevenLabs calls (narration per scene)
3. Apply pricing from `_cookie/_config/pricing.yaml`
4. Add retake buffer (default: 20%)
5. Show resolution alternatives (draft_720p, standard, hd_1080p, 4k)
6. Present breakdown to user for approval

## Stage 2: Real-Time Cost Tracking

Every API-calling workflow must implement cost tracking:

### Before Each API Call
1. Calculate expected cost from pricing.yaml
2. Check cumulative total against budget thresholds

### After Each API Call
1. Log transaction to `_cost/actuals/{episode_id}-actuals.yaml`
2. Update running total
3. Check against thresholds

### Transaction Log Format

### Threshold Behavior

| Threshold | Action |
|-----------|--------|
| Spent < 80% of estimate | `on-track` — continue normally |
| Spent ≥ 80% of estimate | `warning` — notify user, continue |
| Spent ≥ 120% of estimate | `halt` — stop all API calls, present cost report, ask user |
| Per-scene cost > $5 | `warn` — notify user about expensive scene |
| Per-scene cost > $10 | `halt` — require user approval to continue |

### Execution Mode Impact

| Mode | Cost Gates |
|------|-----------|
| `normal` | Full cost approval at every gate |
| `yolo` | Auto-approve but cost gates still fire (warn/halt active) |
| `yolo-uncapped` | Skip all cost gates (must be explicitly enabled) |

## Retake Tracking

When a user requests scene regeneration:

1. Increment retake counter for that scene
2. Log transaction with `is_retake: true` and `retake_reason`
3. Check against `max_retakes_per_scene` (default: 3)
4. If max reached, present options:
   - **Continue** — allow more retakes (resets counter)
   - **Skip** — mark scene as skipped, move on
   - **Adjust prompt** — modify the prompt and try again (counts as new generation, not retake)

### Retake Reason Codes

| Code | Description |
|------|-------------|
| `character_inconsistency` | Character doesn't match reference images |
| `quality_too_low` | Output quality below acceptable threshold |
| `wrong_camera_movement` | Camera movement doesn't match SSD specification |
| `wrong_duration` | Output duration doesn't match request |
| `content_policy` | Content blocked by safety filter |
| `user_preference` | User prefers a different creative direction |
| `audio_artifact` | Audio has artifacts or quality issues |
| `continuity_break` | Visual continuity broken between clips |

## Integration Points

### Workflows That Must Track Costs

| Workflow | Tool | What to Log |
|----------|------|-------------|
| generate-character-refs | Nano Banana | Per-angle image generation |
| generate-frames | Nano Banana | First/last frame generation |
| generate-video-clips | VEO | Per-clip video generation |
| generate-narration | ElevenLabs | Per-scene narration generation |

### Cost Directories

```
_cost/
├── config.yaml            # Thresholds, buffer %, max retakes
├── estimates/             # Pre-generation estimates
│   └── {episode_id}-estimate.yaml
├── actuals/               # Real-time transaction logs
│   └── {episode_id}-actuals.yaml
└── reports/               # Final cost reports
    └── {episode_id}-final-report.yaml
```
